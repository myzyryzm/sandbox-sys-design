# Authoring a sandbox gRPC contract

You are operating inside the "Distributed Systems Sandbox" web app. Each system lives in
`systems/<id>/`; your session's system prompt names the specific `<id>` and inlines its
current `manifest.json`. The web frontend runs under `npm run dev` and reads these files
live, so **never run `./start.sh`** — it tears down the dev server you're attached to.

A **contract** is a `.proto` *service*: a set of RPC methods + their request/response message
types. It is **not** a connection and **not** tied to any pair of nodes — roles (server/client)
and targets are decided later, per service, by [[sandbox-grpc-attach]]. So a contract is a
**per-system registry**, not a diagram node. Authoring a contract generates its implementation
**once** as a single shared servicer that every attaching service in this system imports — this
is what stops a homogeneous fleet from drifting. Differences between nodes must come from
**config** (targets, ports, starting state) or an explicit override, never from divergent
generated code. The web app's "＋ gRPC contract" button drives this flow (`POST /api/grpc-contracts`
persists the method to the registry, then launches this session); by hand, reproduce the same shape.

## The places a contract lives (working dir is the repo root)

Everything is under `systems/<id>/grpc/`, so a contract is fully self-contained and the system
stays independently restartable:

1. `systems/<id>/grpc/<Contract>.proto` — the contract. `<Contract>` is **PascalCase** (e.g.
   `ChunkTransfer`) and is the proto `service` name, the file stem, and the registry key.
2. `systems/<id>/grpc/<Contract>_pb2.py` and `<Contract>_pb2_grpc.py` — generated by `protoc`.
   **Never hand-edit these.**
3. `systems/<id>/grpc/<Contract>_servicer.py` — the **single shared servicer** for the whole
   system: a subclass of the generated `<Contract>Servicer` whose method bodies implement the
   behavior described in the instruction text. Generated **once**; do not re-infer it per attach.
4. `systems/<id>/grpc/_registry.json` — contract metadata the modal reads back:
   ```json
   { "contracts": {
     "ChunkTransfer": {
       "instruction": "the original free-text that drove the implementation",
       "methods": [{ "name": "GetChunk", "request": {"index":"int32"}, "response": {"data":"bytes"}, "responseStreaming": false }],
       "conversationId": "<uuid>", "createdAt": "<iso>" } } }
   ```
   The web app writes/updates this when you author from the modal — keep `instruction` accurate,
   since re-opening a contract shows it as the provenance of the generated behavior.

### Conventions (match these — attach + delete depend on them)
- `<Contract>` PascalCase; RPC methods PascalCase (`GetChunk`); message fields snake_case.
- **Field numbers are sequential and never reused within a message** — proto requires an explicit
  number per field; assign `1, 2, 3, …` in declaration order and never renumber an existing field
  (appending a new field gets the next unused number). This is the single most important proto rule.
- One message per request and per response, named `<Method>Request` / `<Method>Response`.
- A streaming response is `returns (stream <Method>Response)`; otherwise `returns (<Method>Response)`.
- If the modal passed request/response fields, use them verbatim; if it passed **none**, infer a
  reasonable message shape from the instruction text.

## proto skeleton

```proto
syntax = "proto3";

package <id>;            // the system id; keeps generated packages distinct

service <Contract> {
  rpc GetChunk (GetChunkRequest) returns (GetChunkResponse);
  rpc Subscribe (SubscribeRequest) returns (stream SubscribeResponse);
}

message GetChunkRequest  { int32 index = 1; }
message GetChunkResponse { bytes data = 1; bool last = 2; }
```

Adding a method = append its `rpc` line + its two `message`s (new field numbers start at 1
**within each new message**).

## Generate the code (you run INSIDE the web app's dev server)

Run `protoc` via Python's `grpc_tools` so the output matches the runtime. Do it inside a
container that has `grpcio-tools` (any service image will once its `requirements.txt` includes
`grpcio-tools`; or use a throwaway `python:3.12-slim`). From the repo root:

```
python -m grpc_tools.protoc -I systems/<id>/grpc \
  --python_out=systems/<id>/grpc --grpc_python_out=systems/<id>/grpc \
  systems/<id>/grpc/<Contract>.proto
```

This (re)writes `<Contract>_pb2.py` and `<Contract>_pb2_grpc.py`. Then write/refresh the shared
servicer `systems/<id>/grpc/<Contract>_servicer.py`:

```python
from . import <Contract>_pb2 as pb2
from . import <Contract>_pb2_grpc as pb2_grpc

class <Contract>Servicer(pb2_grpc.<Contract>Servicer):
    async def GetChunk(self, request, context):
        # behavior derived from the instruction text + field defs
        return pb2.GetChunkResponse(data=b"", last=True)
```

The servicer runs **in the attaching service's process** (a `grpc.aio` server started in the
FastAPI lifespan on port `50051` — see [[sandbox-grpc-attach]]). gRPC call metrics are exported
through the service's existing `/metrics` (a server interceptor incrementing counters in the same
`prometheus_client` default registry), so **no new Prometheus scrape job is needed** — the
per-service job already collects them. Keep method bodies behavior-only here; **do not build the
chunk-transfer/distribution simulation** (timing, bitmaps, backpressure) unless explicitly asked.

## Record the metadata

Ensure `systems/<id>/grpc/_registry.json` has the contract with its `methods` and the
**original instruction text** (the modal already writes this on `POST /api/grpc-contracts`;
if authoring by hand, write it yourself). This is what the contracts modal lists and re-opens.

## Verify

1. The generated files import cleanly, e.g.:
   ```
   docker run --rm -v "$PWD/systems/<id>/grpc":/g -w /g python:3.12-slim \
     sh -c "pip install -q grpcio && python -c 'import <Contract>_pb2, <Contract>_pb2_grpc'"
   ```
2. `_registry.json` lists the contract + method, and the contracts modal shows the instruction
   text when the contract is re-opened.
3. There is exactly **one** `<Contract>_servicer.py` (shared) — no per-service copies (those are
   only created as explicit overrides by [[sandbox-grpc-attach]]).

## Upload an existing `.proto`

The contracts modal also accepts a whole `.proto` (the **Upload .proto** tab). It POSTs
`/api/grpc-contracts` with `{ system, protoFile, instruction, conversationId }` (instead of the
per-method fields). The backend **validates the file with the real `protoc`** before anything is
created, so an incomplete/invalid proto fails the create with the exact compiler message and
**nothing is registered**. "Complete" means all of:

- declares `syntax = "proto3";`;
- is **self-contained** — no `import` statements (a contract is one file);
- has **exactly one** `service <Name> { … }`, `<Name>` PascalCase, with **≥1 `rpc`**;
- **compiles** under `protoc` (this is what catches undefined/dangling message types like a
  `ChunkFrame` that references a missing `ChunkHeader`, duplicate/missing field numbers, syntax
  errors).

On success the backend writes `systems/<id>/grpc/<Name>.proto` **verbatim** (file stem = service
name = registry key) and registers every rpc method, then launches **this** session. The proto is
already written and already passed protoc — so **do not re-author or rewrite it**. Your job for an
upload is only to:

1. run `protoc` (command above) to generate `<Name>_pb2.py` / `<Name>_pb2_grpc.py`, and
2. write the single shared `<Name>_servicer.py` implementing the uploaded methods (behavior from
   the description, or inferred from the proto if none was given).

By hand (no web app): validate first with the same throwaway-container protoc the Verify section
uses; only if it compiles, drop the `.proto` into `systems/<id>/grpc/`, add the contract to
`_registry.json`, and generate the bindings + shared servicer.

## Update / delete

- **Add or change a method**: append/replace the `rpc` + messages in the `.proto` (new field
  numbers, never reused), re-run `protoc`, update the shared servicer body, and upsert the method
  in `_registry.json`.
- **Delete a contract**: remove `systems/<id>/grpc/<Contract>.proto`, `<Contract>_pb2.py`,
  `<Contract>_pb2_grpc.py`, `<Contract>_servicer.py`, and the `_registry.json` entry, then scrub
  the contract from every service's manifest `grpc` block (`servers`/`clients`/`overrides`) and
  unwire it from those services' code (the web app's contract-delete does the registry/manifest
  scrub; the code unwiring is yours). Any attached service must be rebuilt afterward.
